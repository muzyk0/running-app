# AI Training Schema Draft

## Статус

Это черновой вариант схемы AI-ответа для MVP.

Важно:

- это не финальный контракт
- схема почти наверняка будет меняться
- этот файл нужен как стартовая точка для `ralphex --plan`, а не как жесткое ограничение на будущую архитектуру

Если `ralphex` увидит более удачную структуру для MVP, он может предложить изменения, но без ненужного усложнения.

Связанные документы:

- `docs/planning/product-brief.md`
- `docs/planning/ralphex-plan-request.md`
- `docs/planning/post-mvp-roadmap.md`
- `docs/planning/architecture-notes-draft.md`

## Зачем нужен этот draft

Сейчас задача не в том, чтобы заморозить доменную модель, а в том, чтобы:

- уйти от свободного текста в ответах AI
- дать backend и Android-клиенту понятную стартовую структуру
- упростить валидацию ответа модели
- упростить сохранение, редактирование и запуск тренировки

## Принцип проектирования

На первом этапе стоит проектировать так:

- схема достаточно конкретная для MVP
- модель данных версионируемая
- список полей расширяемый
- Android storage, API-модели и backend-валидация не должны быть слишком жестко привязаны только к этой версии

Практический вывод:

- закладывать `schema_version`
- хранить явные DTO/transport-модели отдельно от внутренних доменных моделей
- оставлять место для будущих полей шага тренировки

## Предлагаемая структура ответа

### Вариант top-level envelope

```json
{
  "schema_version": "draft-1",
  "training": {
    "title": "Интервальная тренировка для новичка",
    "summary": "Чередование легкого бега и ходьбы",
    "goal": "Адаптация к беговой нагрузке",
    "estimated_duration_sec": 1680,
    "disclaimer": "Оцените самочувствие перед началом тренировки.",
    "steps": [
      {
        "id": "step-1",
        "type": "warmup_walk",
        "duration_sec": 300,
        "voice_prompt": "Начинаем с разминки. Идите быстрым шагом 5 минут."
      },
      {
        "id": "step-2",
        "type": "run",
        "duration_sec": 120,
        "voice_prompt": "Переходим на легкий бег на 2 минуты."
      },
      {
        "id": "step-3",
        "type": "walk",
        "duration_sec": 300,
        "voice_prompt": "Переходим на ходьбу на 5 минут."
      },
      {
        "id": "step-4",
        "type": "cooldown_walk",
        "duration_sec": 180,
        "voice_prompt": "Заканчиваем тренировку спокойной ходьбой."
      }
    ]
  }
}
```

## Сущности draft-схемы

### 1. Envelope

Top-level объект ответа от backend или от AI-нормализатора.

Поля:

- `schema_version`: версия черновой схемы
- `training`: сама тренировка

### 2. Training

Предлагаемые поля:

- `title`: короткое имя тренировки
- `summary`: краткое описание
- `goal`: цель текущей тренировки
- `estimated_duration_sec`: суммарная длительность
- `disclaimer`: текстовое предупреждение для UI
- `steps`: массив шагов

### 3. TrainingStep

Поля MVP:

- `id`: стабильный идентификатор шага внутри тренировки
- `type`: тип шага
- `duration_sec`: длительность шага в секундах
- `voice_prompt`: текст голосовой подсказки

## Предлагаемые типы шагов

Это тоже черновой список, а не финальный enum:

- `warmup_walk`
- `run`
- `walk`
- `cooldown_walk`
- `rest`

Если в MVP окажется, что достаточно меньшего набора, можно упростить до:

- `run`
- `walk`
- `warmup`
- `cooldown`

## Какие поля считать обязательными в MVP

Для top-level:

- `schema_version`
- `training`

Для `training`:

- `title`
- `estimated_duration_sec`
- `steps`

Для `step`:

- `id`
- `type`
- `duration_sec`
- `voice_prompt`

## Что можно считать опциональным в MVP

- `summary`
- `goal`
- `disclaimer`

Если эти поля не придут, приложение все равно должно уметь показать и запустить тренировку.

## Базовые правила валидации

Backend должен проверять и при необходимости нормализовать ответ модели до сохранения и отдачи клиенту.

Минимальные правила:

- `steps` не пустой массив
- у каждого шага `duration_sec > 0`
- сумма `duration_sec` по шагам совпадает с `estimated_duration_sec` или backend сам пересчитывает это поле
- все `id` шагов уникальны в рамках тренировки
- `type` входит в разрешенный список или маппится к нему
- `voice_prompt` не пустой

## Что сознательно не включаем в этот draft

Пока не добавляем в схему как обязательные поля:

- темп шага
- дистанцию шага
- RPE
- пульс
- зоны пульса
- целевые калории
- cadence
- сложные правила прогрессии между тренировками

Это должно остаться в post-MVP scope.

## Точки расширения на будущее

Схема должна безболезненно пережить добавление:

- `target_pace`
- `target_distance_m`
- `target_rpe`
- `target_hr_zone`
- `notes`
- `audio_cue_asset`
- `repeat_group`
- `tags`

## Что важно для `ralphex --plan`

План должен исходить из того, что:

- это рабочий draft, а не замороженный стандарт
- нужно предусмотреть эволюцию схемы без болезненной миграции всего приложения
- в MVP достаточно поддержки текущего минимального набора полей
- backend отвечает за нормализацию и валидацию AI-ответа
- Android-клиент должен уметь хранить, показывать, редактировать и запускать тренировку по этой стартовой структуре

## Практическая рекомендация

Если делать pragmatically, то на первом этапе стоит:

1. принять этот draft как стартовую transport-схему
2. не привязывать к нему слишком жестко внутреннюю доменную модель
3. предусмотреть версионирование и future-proof миграции
4. оставить расширения шагов тренировок на post-MVP
