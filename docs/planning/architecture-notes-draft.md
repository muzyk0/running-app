# Architecture Notes Draft

## Статус

Это черновые архитектурные заметки для MVP.

Важно:

- это не финальная архитектура
- структура может и будет меняться по мере реализации
- файл нужен как стартовая рамка для `ralphex --plan`, а не как ограничение, в которое нужно любой ценой втиснуть проект

Если в ходе планирования или реализации появится более прагматичный вариант, его можно принимать.

## Зачем нужен этот draft

Сейчас важно дать планированию стартовую форму:

- как логически разделить Android-клиент и backend
- где проходят основные границы ответственности
- как не переусложнить green field MVP
- как заложить точки расширения без ранней overengineering

## Архитектурные принципы для MVP

Предлагаемые принципы:

- сначала рабочий MVP, потом усложнение
- минимально достаточная модульность
- явное разделение transport-моделей, storage-моделей и domain-моделей там, где это действительно окупается
- backend отвечает за генерацию, нормализацию и валидацию AI-ответа
- Android-приложение остается основным владельцем пользовательских данных, тренировок и истории активности
- все draft-контракты должны допускать эволюцию

## Черновая структура репозитория

Для green field MVP разумная стартовая структура может быть такой:

```text
/
  android-app/
  backend/
  docs/
    ralphex.md
    planning/
      product-brief.md
      ralphex-plan-request.md
      ai-training-schema-draft.md
      architecture-notes-draft.md
      post-mvp-roadmap.md
```

Почему так:

- Android и backend сразу физически разделены
- planning-документы не смешиваются с общими docs
- монорепо достаточно для MVP и не создает лишней операционной сложности

## Android-приложение: стартовый draft

### Базовый стек

Рабочая стартовая гипотеза:

- Kotlin
- Jetpack Compose
- Navigation Compose
- Room для основной локальной модели
- DataStore только для мелких флагов и настроек, если понадобится
- Fused Location Provider для GPS

### Границы подсистем

В MVP логично видеть такие функциональные области:

- `profile`
- `training`
- `activity`
- `ai`
- `core`

Что они значат:

- `profile`: данные пользователя и дисклеймер
- `training`: локальные тренировки, просмотр, редактирование, запуск
- `activity`: свободный бег, GPS-трекинг, запись маршрута и метрик
- `ai`: запрос генерации тренировки и маппинг ответа backend
- `core`: навигация, storage, common utilities, DI

### Модульность Android

Для MVP лучше не начинать с большого числа Gradle-модулей.

Прагматичный старт:

- один Android app module
- разделение по пакетам и слоям внутри приложения

Если проект начнет быстро расти, потом можно вынести отдельные feature/data/core modules.

Это безопаснее, чем переусложнять сборку с первого дня.

### Черновой набор экранов

Минимальный набор UI-сценариев:

- onboarding или первый запуск
- профиль пользователя
- список тренировок
- экран генерации тренировки через AI
- просмотр и редактирование тренировки
- экран выполнения тренировки
- экран свободного бега
- история активностей

## Локальные данные Android

В первой версии имеет смысл хранить локально:

- профиль пользователя
- сохраненные тренировки
- историю выполненных тренировок
- историю свободных пробежек
- GPS-точки маршрута для активностей

Черновой принцип хранения:

- backend не является source of truth для этих данных в MVP
- мобильное приложение само отвечает за локальное состояние

## Backend: стартовый draft

### Роль backend в MVP

Backend нужен для одной основной задачи:

- принять запрос на генерацию тренировки
- собрать prompt из входных данных
- вызвать AI-провайдер
- провалидировать и нормализовать ответ
- вернуть структурированную тренировку клиенту

### Что backend не обязан делать в MVP

- хранить пользовательские аккаунты
- синхронизировать тренировки между устройствами
- быть полноценной системой учета всей активности
- хранить большую доменную базу данных

Из этого следует прагматичный вывод:

- backend можно делать преимущественно stateless
- база данных на backend для MVP не обязательна, если не появится отдельная операционная причина

### Внутренняя структура backend

Черновое деление:

- `api` или `http`: HTTP handlers и request/response DTO
- `service`: orchestration логики генерации
- `provider`: интерфейс AI-провайдера и реализации
- `prompt`: сборка системного и пользовательского prompt
- `schema`: валидация и нормализация training schema
- `config`: конфигурация и runtime settings

### Go project layout

Для backend можно использовать репозиторий `golang-standards/project-layout` как ориентир по структуре каталогов.

Но важно трактовать его именно как reference, а не как обязательный стандарт:

- это не официальный стандарт Go
- сам репозиторий предупреждает, что для простого или PoC-проекта такая структура может быть overkill
- для MVP нужно брать только те части layout, которые реально помогают, а не копировать весь шаблон целиком

Практический вывод для этого проекта:

- с высокой вероятностью будут полезны `cmd`, `internal`, `configs`, `deployments`
- `pkg`, `api`, `scripts`, `test` и другие каталоги стоит добавлять только при реальной необходимости
- backend-структура должна оставаться компактной, пока проект не усложнился

### Провайдеры backend

Для MVP:

- основная реализация: `codex` CLI provider

На будущее:

- OpenAI provider
- Anthropic provider
- другие API provider implementations

Важный принцип:

- все провайдеры должны быть спрятаны за единым интерфейсом
- transport-контракт backend для мобильного клиента не должен зависеть от конкретного AI-провайдера

## Черновой API-контракт между Android и backend

На MVP можно стартовать с минимального HTTP JSON API.

Базовые endpoints:

- `GET /healthz`
- `POST /v1/trainings/generate`

### Черновой запрос на генерацию

Идея запроса:

- snapshot профиля пользователя
- цель или текстовый запрос пользователя
- дополнительные поля контекста
- возможно, технические поля вроде версии клиента

### Черновой ответ

Ответ:

- training envelope из `docs/planning/ai-training-schema-draft.md`
- возможно, service metadata, если это реально нужно и не захламляет MVP

## Разделение ответственности

### Android

Android отвечает за:

- ввод и хранение профиля
- хранение тренировок
- редактирование тренировки
- запуск тренировки
- GPS-трекинг и локальную историю
- UX дисклеймера

### Backend

Backend отвечает за:

- prompt assembly
- вызов AI-провайдера
- нормализацию AI-ответа
- валидацию schema draft
- возврат клиенту предсказуемой структуры

## Точки, где важно не зацементировать решение слишком рано

Особенно важно оставить пространство для изменений в:

- training schema
- наборе типов шагов
- формате prompt assembly
- способе хранения тренировок в Android
- числе модулей Android-проекта
- наборе backend провайдеров

## Практический совет для `ralphex --plan`

План должен исходить из таких ограничений:

- архитектура стартовая и может меняться
- не нужно вводить сложную модульность раньше времени
- Android и backend должны быть разделены по ответственности с самого начала
- backend можно планировать как stateless-сервис для MVP
- draft schema и draft architecture должны эволюционировать без дорогостоящих переделок

## Что полезно уточнить позже

Эти вопросы пока можно не считать blocker для MVP, но их придется пересмотреть:

- нужен ли backend storage для логов, rate limits или аудита генераций
- стоит ли выделять Android feature-модули сразу после MVP
- хранить ли профиль полностью в Room или частично в DataStore
- нужна ли фоновая работа и foreground service для длительного трекинга с экранами блокировки
- как далеко зайдет кастомизация аудио-подсказок
