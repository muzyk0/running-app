# Running App MVP Request For `ralphex --plan`

## Цель

Подготовить технический план реализации green field проекта: нативного Android-приложения для бега с локальным хранением данных, GPS-трекингом активности и AI-генерацией интервальных тренировок через отдельный backend.

Итог `ralphex --plan` должен дать последовательный roadmap реализации MVP, который потом можно будет выполнять по задачам.

## Что строим

Продукт состоит из двух частей:

1. Android-приложение на Kotlin, только для Android, без iOS и web.
2. Backend для AI-генерации тренировок, предпочтительно на Go.

Язык интерфейса приложения: русский.

## Зафиксированные продуктовые решения

### Платформа и клиент

- только Android
- нативное приложение
- Kotlin
- локальная база данных на устройстве
- локальный пользовательский профиль
- работа без обязательного аккаунта

### Данные профиля пользователя

Обязательные поля:

- рост
- вес
- пол
- возраст
- желаемое количество тренировок в неделю
- уровень подготовки
- ограничения и травмы
- цель тренировок

Дополнительно:

- свободные дополнительные поля, которые включаются в AI prompt

### Функции тренировок

- пользователь может запросить одну тренировку через AI
- тренировка приходит в структурированном формате
- тренировка сохраняется локально
- тренировку можно просмотреть
- тренировку можно отредактировать вручную
- тренировку можно удалить
- тренировку можно запустить
- во время выполнения есть аудиосигналы для переключения этапов

### Режимы активности

В MVP нужны оба режима:

- выполнение сохраненной тренировки
- свободный бег

### Трекинг активности в MVP

В первой версии нужно сохранять только:

- маршрут
- дистанцию
- длительность
- средний темп

Сознательно не включать в MVP:

- калории
- пульс
- сплиты по километрам
- автопаузу

### AI в MVP

- AI генерирует только одну тренировку за запрос
- недельные планы и многонедельная прогрессия не входят в MVP

### Контракт шага тренировки в MVP

Каждый шаг тренировки пока описывается только такими полями:

- тип шага
- длительность
- голосовая подсказка

Дополнительные цели шага пока не нужны:

- темп
- дистанция шага
- RPE
- пульс

### Safety

В MVP нужен только дисклеймер о том, что приложение не является медицинской рекомендацией.

Автоматические блокировки рискованных запросов пока не требуются.

## Backend и провайдеры генерации

### Основной вариант генерации

- backend запускает `codex` CLI
- backend передает prompt
- backend получает ответ и преобразует его в структурированную модель для клиента
- можно ориентироваться на идеи и подходы из `umputun/ralphex`, если это ускорит реализацию

### Архитектурное ограничение

Нужно сразу спроектировать backend так, чтобы позже можно было добавить провайдеров через API-ключи:

- OpenAI
- Anthropic
- другие совместимые провайдеры

### Варианты запуска backend

Нужно поддержать оба способа:

- daemon/server на Linux VPS
- контейнерный запуск для локальной разработки и сервера

## Ожидания от плана

`ralphex --plan` должен:

- предложить архитектуру Android-приложения
- предложить архитектуру backend
- определить контракт между мобильным приложением и backend
- определить JSON-структуру AI-ответа для тренировки
- разложить MVP на последовательные технические задачи
- учитывать, что это green field проект
- учитывать, что сначала нужен рабочий MVP, а не максимальный функционал

Стартовый черновик AI-контракта вынесен в `docs/planning/ai-training-schema-draft.md`.
Это именно draft, а не финальная схема, поэтому архитектура должна допускать ее развитие без болезненной переделки всего проекта.

Стартовые архитектурные заметки вынесены в `docs/planning/architecture-notes-draft.md`.
Это тоже draft, а не финальная архитектура.

## Что желательно отразить в задачах

План желательно разбить на подсистемы:

1. bootstrap проекта и репозиторная структура
2. Android-приложение: архитектурный каркас, навигация, storage, permissions
3. профиль пользователя
4. модель тренировки и локальное хранение
5. UI для просмотра, редактирования, удаления и запуска тренировки
6. выполнение тренировки с таймером и аудиосигналами
7. GPS-трекинг для тренировки и свободного бега
8. backend API
9. интеграция backend с `codex` CLI
10. валидация и нормализация AI-ответа
11. интеграция Android-клиента с backend
12. базовые тесты и smoke checks

## Технические допущения, которые можно использовать как стартовые

Если в проекте пока ничего нет, допустимо предложить такие дефолты:

- Android UI: Jetpack Compose
- локальная БД: Room
- DI: Hilt или другой прагматичный стандартный вариант
- GPS: Fused Location Provider
- backend API: HTTP JSON
- backend packaging: Dockerfile + локальный запуск без контейнера
- для backend можно ориентироваться на `golang-standards/project-layout`, но без механического копирования всей структуры

Если `ralphex` предложит лучшие альтернативы для MVP, это допустимо, но без неоправданного усложнения.

## Не цели первой версии

- iOS
- web
- облачная синхронизация
- обязательная регистрация и логин
- социальные функции
- расширенная аналитика
- планы тренировок на недели вперед
- продвинутая спортивная или медицинская safety-логика

## Предпочтение по результату

Нужен такой план, который можно будет выполнять итеративно, начиная с минимального каркаса проекта и заканчивая рабочим MVP, пригодным для локального использования и дальнейшего развития.

Post-MVP идеи и отложенные направления вынесены отдельно в `docs/planning/post-mvp-roadmap.md` и не должны попадать в обязательный scope первой версии.

