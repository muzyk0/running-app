# Running App Product Brief

## Статус

Черновой brief для подготовки к запуску `ralphex --plan`.
Сейчас цель не в реализации, а в фиксации требований, ограничений и открытых решений.

Связанный post-MVP roadmap: `docs/planning/post-mvp-roadmap.md`.
Связанный draft AI-схемы: `docs/planning/ai-training-schema-draft.md`.
Связанные draft архитектурные заметки: `docs/planning/architecture-notes-draft.md`.

## Главный результат этого этапа

Сформировать достаточно конкретный MVP scope, чтобы `ralphex --plan` мог разложить работу на технические задачи без лишних догадок.

## Уже зафиксированные решения

### Продукт

- Приложение для бега с трекингом расстояния, маршрута и AI-генерацией тренировок.
- Язык интерфейса приложения: русский.
- Платформа: только Android.
- Форм-фактор: нативное Android-приложение.
- Язык мобильного приложения: Kotlin.
- Локальное хранение данных в базе на устройстве.

### Данные пользователя

Пользователь заполняет профиль, который как минимум включает:

- рост
- вес
- пол
- возраст
- желаемое количество тренировок в неделю
- уровень подготовки
- ограничения и травмы
- цель тренировок
- дополнительные пользовательские поля, которые будут включаться в системный prompt для AI

### Работа с тренировками

- Пользователь запрашивает тренировку через AI.
- В AI передается системный prompt с учетом профиля пользователя и дополнительных полей.
- AI должен возвращать структурированный ответ, пригодный для отображения в интерактивном UI.
- Сгенерированная тренировка сохраняется локально.
- Тренировку можно удалить.
- Тренировку можно редактировать вручную.
- Тренировку можно запускать.
- Во время выполнения должны быть звуковые сигналы для переключения этапов, например бег/ходьба.

### Backend для AI

- Нужен отдельный backend для генерации тренировок.
- Предпочтительный основной язык backend: Go.
- Два варианта запуска:
  - daemon/server на Linux VPS
  - контейнерный запуск для локальной разработки и сервера

### Варианты провайдеров генерации

#### Вариант 1. Основной

- Backend запускает `codex` CLI.
- Backend передает prompt, ждет ответ и отдает результат приложению в структурированном виде.
- За основу можно взять идеи или части кода из `umputun/ralphex`.

#### Вариант 2. Расширяемый

- Оставить возможность позже добавить генерацию через API-ключи провайдеров вроде OpenAI, Anthropic и других.

## Предварительная граница MVP

Чтобы не расползтись на первом проходе, разумно считать MVP таким:

- локальный профиль пользователя
- локальное хранение тренировок
- AI-генерация одной тренировки по запросу
- просмотр, редактирование и удаление тренировки
- запуск тренировки с пошаговыми этапами и аудиосигналами
- базовый GPS-трекинг дистанции и маршрута во время активности
- два режима активности: запуск тренировки и свободный бег
- backend только для генерации тренировок

## Технические предположения на текущий момент

Это пока не финальные решения, а рабочие допущения для будущего плана:

- Android UI вероятнее всего на Jetpack Compose.
- Локальная БД вероятнее всего Room.
- GPS-трекинг вероятнее всего через Fused Location Provider.
- Формат ответа от AI лучше сразу нормализовать в JSON-схему, а не разбирать свободный текст.
- Backend должен иметь абстракцию над провайдером генерации, чтобы `codex` CLI и API-провайдеры подключались через общий интерфейс.

## Уточненные решения по MVP

Ниже решения, которые были подтверждены и теперь считаются частью рабочей постановки.

### 1. Состав MVP трекинга

В первой версии сохраняем только:

- маршрут
- дистанцию
- длительность
- средний темп

Сознательно не включаем в MVP:

- калории
- пульс
- сплиты по километрам
- автопаузу

### 2. Режимы активности

В MVP нужны оба режима:

- выполнение сохраненной тренировки
- свободный бег без тренировки

### 3. Объем AI-функциональности

В первой версии AI генерирует только одну тренировку за один запрос.

Сознательно не включаем в MVP:

- недельные планы
- многонедельную прогрессию
- долгосрочную адаптацию плана

### 4. Контракт шага тренировки

На текущем этапе шаг тренировки должен включать:

- тип шага
- длительность
- голосовую подсказку

Сознательно не включаем в первую версию шага:

- целевой темп
- дистанцию шага
- RPE
- пульс

### 5. Обязательные данные профиля

Обязательные поля профиля пользователя:

- рост
- вес
- пол
- возраст
- желаемое количество тренировок в неделю
- уровень подготовки
- ограничения и травмы
- цель тренировок

Дополнительно:

- произвольные поля пользователя для включения в AI prompt

### 6. Safety для первой версии

Для MVP требуется:

- дисклеймер о том, что приложение не является медицинской рекомендацией

Сознательно не включаем в MVP:

- автоматические блокировки рискованных запросов
- жесткие safety-ограничения на стороне backend

## Открытые вопросы второго приоритета

Ниже только вопросы, которые действительно влияют на будущую декомпозицию задач.

### 1. Как выглядит полный контракт AI-ответа

Нужно определить:

- из каких сущностей состоит тренировка
- какие поля обязательны у тренировки помимо списка шагов
- как кодируются типы шагов
- какие ограничения по длительности и порядку шагов должны проверяться
- кто валидирует ответ модели перед сохранением

Рабочая гипотеза:

- тренировка состоит из шагов
- каждый шаг имеет тип, длительность и голосовую подсказку
- стартовый черновик схемы вынесен в `docs/planning/ai-training-schema-draft.md`

### 2. Нужна ли аутентификация

Если приложение полностью локальное и без синхронизации, то для MVP можно обойтись без:

- регистрации
- логина
- облачного аккаунта

Это сильно уменьшает объем первой версии.

### 3. Насколько глубоко нужен offline-режим

Сейчас предполагается:

- профиль, история и запуск уже сохраненных тренировок работают офлайн
- генерация новой тренировки требует доступа к backend

## Рекомендация для следующего шага

Сейчас важнее всего не обсуждать все возможные функции, а закрыть 3 блока решений:

1. финальный JSON-контракт AI-тренировки
2. границы локального/offline-поведения
3. технический каркас Android-приложения и backend

После этого можно подготовить отдельный файл-запрос для `ralphex --plan`, где уже будут:

- product scope
- архитектурные ограничения
- список подсистем
- последовательность реализации MVP
